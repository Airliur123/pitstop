# Index Rationale

Indexes follow Phase 1 query paths; redundant left-prefix indexes are intentionally avoided.

| Index                                               | Query served                                    |
| --------------------------------------------------- | ----------------------------------------------- |
| `idx_places_location` (SPATIAL)                     | geographic radius and distance candidate lookup |
| `idx_places_status_verification`                    | public candidate eligibility                    |
| `idx_places_deleted_at`                             | soft-delete administration and maintenance      |
| `idx_places_data_freshness`                         | stale-data maintenance                          |
| `idx_places_district_city`                          | manual-area lookup                              |
| `idx_menus_place_main_available_price`              | valid main-item budget hard filter              |
| `uq_menus_place_name`                               | idempotent menu seed/update                     |
| `idx_operating_hours_place_day`                     | current-day opening intervals                   |
| `idx_contributions_status_created`                  | moderation queue ordering                       |
| `idx_place_reports_status_created`                  | report queue ordering                           |
| `idx_audit_target_created`                          | target audit timeline                           |
| `uq_google_form_source_external`                    | inbound submission idempotency                  |
| `idx_refresh_tokens_user_family_revoked`            | family lock and revocation                      |
| `idx_geocoding_contribution`, `idx_geocoding_place` | subject-specific geocoding lookup               |

Unique indexes also enforce active normalized email, provider account, primary category/photo,
place confirmation, category/facility codes, source code, idempotency scope/key, media object keys,
and operating interval keys. Foreign-key support indexes generated by InnoDB are not duplicated
unless a wider query-specific composite index is required.
